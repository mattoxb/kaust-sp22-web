This writeup will teach you how to use a real database. Why would you want to do that?

  • Sometimes a spreadsheet is not powerful enough to store the data you need.
  • A real database allows for queries and data validation.
  • We can use Python to access the database very simply.

  • A database allows information to persist beyond the lifetime of a single program.

    So let’s get started!

Beginning

We are going to use a database called SQLite3. Some of its features:

  • It is very small, and perfect for “embedded databases”. You almost certainly have some SQLite database on your phone.
  • It is full featured. You can enjoy most of the features of a good database system.
  • It does not require a server to run.
  • Because of that, the ndsqlite files it saves to are shareable.

To get started, go to https://repl.it and create a new REPL. Tell it you want an SQLite instance. You will see the usual panel containing a program and another panel containing an interactive shell.

The program panel just has one command: 

select "Hello world!";

You can probably guess what it does.

Try a few other commands in the shell using select: 

> select 10;
10
> select 10 * 30;
300
> select max(10,30);
30
> select min(10,20);
10

Select is boring though without data to select, so we need to make some data.

Tables

Basic Tables

SQL SQL stands for Structured Query Language. is a domain specific language for storing information in a Relational Database More formally, it’s called a Relational Database Management System or RDMS.

The database consists of tables. Each table has a specification called a schema that gives the names and types of the columns in the table. The data lives in the rows of the table, and must match the schema.

Let’s say you want to keep a database of books in your library. We might create one like this: 

create table books (
  title text,
  author text,
  style text,
  year integer,
  pages integer,
  read boolean
);

Well and good, but now we need some data. We can do this using the insert command. 

insert into books values
   ('Meditations','Marcus Aurelius','Philosophy',180,170,true),
   ('The Hobbit','Tolkien','Fantasy',1937,310,true),
   ('The Silmarilian','Tolkien','Fantasy',1977,365,false),
   ('Category Theory','Steve Awodey','Mathematics',2010,311,false),
   ('Conceptual Mathematics','Lawvere','Mathematics',2009,390,false),
   ('Leviathan Wakes','James Corey','Sci Fi',2011,582,false);

Please copy these and put them into your main.sql file and hit Run.

Searching for attributes

You can select for a specific value if you want. 

> select * from books where author='Tolkien';
title            author   style    year  pages  read
---------------  -------  -------  ----  -----  ----

The Silmarilian  Tolkien  Fantasy  1977  365    0

And count them…. 

> select count(*) from books where author='Tolkien';
count

Suppose you want a short book. 

> select *,min(pages) from books;
title        author           style       year  pages  read  min(pages)

Okay, how about a short math book.

Linking Tables

The next thing we want to do is link tables together. We can join tables using a select. Suppose we have a table for books and a table for authors. 

create table books (
  title text,
  author_fname text,
  author_lname text,
  year integer,
  pages integer,
  style text,
  read boolean
);
create table authors (
  fname text,
  lname text,
  born integer,
  died integer);

We can add some data now the previous way. 

insert into books values
   ('Meditations','Marcus','Aurelius','Philosophy',170,180,true),
   ('The Hobbit','J.R.R.','Tolkien','Fantasy',1937,310,true),
   ('The Silmarilian','J.R.R.','Tolkien','Fantasy',1977,365,false),
   ('Category Theory','Steve','Awodey','Mathematics',2010,311,false),
   ('Conceptual Mathematics','F. William','Lawvere','Mathematics',2009,390,false),
   ('Leviathan Wakes','James','Corey','Sci Fi',2011,582,false);
insert into authors values
   ('Marcus','Aurelius',121,180),
   ('J.R.R.','Tolkien',1892,1973),
   ('Steve','Awodey',1959,NULL),
   ('F. William','Lawvere',1937,NULL),
   ('Stephen','Schanuel',1933,2014),
   ('James','Corey',2011,NULL);

Suppose we want to have a list of titles, author last name, and author birth year. We can use select to combine the data from two tables. Here’s an example. 

> select books.title, authors.lname, authors.born
    from books, authors
   where authors.lname = books.author_lname and authors.fname = books.author_fname;
title                   lname     born
----------------------  --------  ----
Meditations             Aurelius  121
The Hobbit              Tolkien   1892
The Silmarilian         Tolkien   1892
Category Theory         Awodey    1959
Conceptual Mathematics  Lawvere   1937
Leviathan Wakes         Corey     2011

You see that the column names are prefixed by the table they came from.

Problems with this approach

There are a few things wrong with this.

  • Data is duplicated! We have to repeat the first name and last name columns. This is inefficient, and error prone. What if you misspell something?
  • Are you wondering who Stephen Schanuel is? He is the second author of Conceptual Mathematics.
create table books (
  id integer,
  title text,
  author_id integer,
  year integer,
  pages integer,
  style text,
  read boolean
);
create table authors (
  id integer,
  fname text,
  lname text,
  born integer,
  died integer);

Foreign Key

create table authors (
  id integer,
  fname text,
  lname text,
  born integer,
  died integer);
create table books (
  id integer,
  title text,
  author_id integer references authors(id),
  year integer,
  pages integer,
  style text,
  read boolean
);

Join Tables

create table authors (
  id integer,
  fname text,
  lname text,
  born integer,
  died integer);
create table books (
  id integer,
  title text,
  year integer,
  pages integer,
  style text,
  read boolean
);
create table book_authors (
  book_id integer references books(id),
  author_id integer references authors(id)
);

Views

Typing that command to link together tables is going to be a pain. We can “save” the selection as its own table. 

create view bookborn as select books.title, authors.lname, authors.born
    from books, authors
   where books.author_id = authors.id;
